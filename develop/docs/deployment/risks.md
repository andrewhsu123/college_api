# 风险和应对

## 技术风险

### 风险1：JSON 字段查询性能问题

**影响**：如果频繁按机构查询角色，JSON 字段无法使用索引

**应对**：
- 使用 `role_has_departments` 关联表辅助查询
- 缓存角色管辖范围
- 如果性能仍不满足，考虑将 JSON 字段拆分为关联表

### 风险2：缓存一致性问题

**影响**：缓存和数据库数据不一致

**应对**：
- 使用延迟双删策略
- 设置合理的缓存过期时间
- 提供手动刷新缓存接口

### 风险3：数据库主从延迟

**影响**：读从库可能读到旧数据

**应对**：
- 实时性要求高的读操作走主库
- 监控主从延迟，超过阈值告警
- 使用半同步复制

### 风险4：ElasticSearch 数据同步延迟

**影响**：搜索结果不是最新的

**应对**：
- 提示用户搜索结果可能有延迟
- 监控 MQ 消息堆积
- 提供强制刷新功能

## 业务风险

### 风险1：数据迁移失败

**影响**：历史数据丢失或错误

**应对**：
- 充分测试迁移脚本
- 先在测试环境验证
- 保留原始数据备份
- 提供数据校验工具

### 风险2：权限计算错误

**影响**：用户看到不该看的数据，或看不到该看的数据

**应对**：
- 充分的单元测试和集成测试
- 提供权限调试工具
- 灰度发布，逐步放量
- 提供权限审计日志

### 风险3：性能不达标

**影响**：接口响应慢，影响用户体验

**应对**：
- 提前压测，发现性能瓶颈
- 准备降级方案
- 监控关键指标，及时优化
